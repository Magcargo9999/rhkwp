### 문제 1. Stakeholder란 무엇이고, Stakeholder가 증가할 경우에 발생하는 문제점에 대해서 "추상화와 모델링"이 어떠한 효과가 있는지 설명하시오.

* **Stakeholder**는 하나의 프로젝트에 이해관계를 가진 **이해관계자**를 말합니다.
* 한 프로젝트에는 목표를 제시하는 고객, 제품을 사용하는 사용자, 요구사항을 정리하는 기획자, 기능을 구현하는 개발자, 그리고 배포와 유지보수를 담당하는 운영자 등 수많은 이해관계자가 존재합니다. 
* **문제점:** 이해관계자가 늘어날수록 소통 채널의 수가 기하급수적으로 증가하여 원활한 소통이 어려워집니다. 특히 서로 다른 관점으로 프로젝트를 바라보기 때문에 소통이 어긋나면 결국 고객의 의도와는 다른 결과물이 나올 수 있습니다.

![stakeholder_complexity](https://github.com/user-attachments/assets/85925a22-aff3-4284-a4e9-73db1e147879)

* **추상화와 모델링의 효과**
    * **추상화:** 본질적이지 않은 부분을 제거하고 핵심적인 부분만 추출하여 단순화함으로써, Stakeholder들이 프로젝트의 핵심 내용을 쉽게 이해하도록 돕습니다.
    * **모델링:** 추상화된 부분을 바탕으로 설계 도면을 시각화하여, 모든 이해관계자가 **동일한 관점**에서 시스템을 이해하고 소통할 수 있게 합니다.

### 문제 2. 소프트웨어의 개발 기간과 이용 기간 장기화에 의해 발생하는 문제점에 대해서 "추적 가능성"이 어떠한 효과가 있는지 설명하시오.

* **개발 기간 장기화 시 문제점:** 개발 기간이 길어지면 소프트웨어의 개발흐름을 어떠한 기준으로 왜 그렇게 프로그래밍했는지 파악하기 어려워집니다. 특히 초기 개발자가 퇴사할 경우 코드의 핵심 개념과 히스토리를 아는 인력이 사라져 유지보수에 큰 어려움을 겪게 됩니다.
* **이용 기간 장기화 시 문제점:** 장기 이용 시 반복적인 수정으로 인해 코드가 복잡해지며, 처음 설계 단계의 형태와 실제 운영 중인 프로그램의 형태가 크게 달라져 시스템 파악이 어려워집니다.
* **추적 가능성(Traceability)의 효과:** * 추적 가능성은 개발 과정에서 발생하는 결정 사항과 변경 사항을 문서화하여 요구사항부터 코드까지 연결하는 것입니다. 
    * 이를 통해 설계 도면과 히스토리를 명확히 남겨두면, 시간이 흐른 뒤에도 개발 의도를 파악할 수 있어 정보 유실 문제를 해결할수있습니다. 
    * 결과적으로 디버깅이나 코드 수정 시 영향 범위를 빠르게 파악하여 투입되는 **리소스(시간과 비용)**를 획기적으로 절감할 수 있습니다.
